# План реализации: Голосовой чат-бот (Electron + TS)

Стек: **Electron + electron-vite + React + TypeScript**, SQLite (better-sqlite3), Vitest (+ React Testing Library). Платформа v1: **macOS (Apple Silicon)**.

Принцип работы: идём строго по шагам. Каждый шаг заканчивается **гейтом**: `typecheck + build + тесты зелёные + приложение запускается`. Не переходим к следующему шагу, пока гейт не пройден.

Объём v1 (по решению заказчика): **UI + claude CLI + Whisper — реальные**. Диаризация и TTS — заглушки с чистыми интерфейсами (реальная реализация — следующий этап).

---

## Definition of Done для каждого шага (гейт)
1. `npm run typecheck` — без ошибок.
2. `npm run test` — все тесты зелёные.
3. `npm run build` — сборка проходит.
4. `npm run dev` / собранное приложение — запускается без ошибок в консоли main/renderer.
5. Ручная/скриптовая проверка результата шага (smoke).

---

## Шаг 0 — Каркас и тулчейн
- Инициализация проекта: `package.json`, TypeScript, electron-vite, React, ESLint/Prettier, Vitest.
- Структура: `src/main`, `src/preload`, `src/renderer`, `src/shared`.
- Пустое окно Electron, которое запускается и показывает renderer.
- **Тесты:** тривиальный smoke-тест Vitest (`sanity.test.ts`), проверка что тест-раннер работает.
- **Гейт:** `npm run dev` открывает пустое окно; `npm run test` зелёный.

## Шаг 1 — Shared-типы и машина состояний (чистая логика)
- `src/shared/types.ts`: `Message`, `Segment`, `Speaker`, `Conversation`, `Settings`, `VoiceState`.
- `src/shared/stateMachine.ts`: чистый редьюсер переходов `idle → listening → transcribing → thinking → speaking → idle` + barge-in + запреты недопустимых переходов.
- **Тесты:** unit-тесты всех валидных/невалидных переходов, barge-in из speaking в listening, сброс.
- **Гейт:** тесты покрывают все переходы, зелёные.

## Шаг 2 — Слой хранения (SQLite)
- `src/main/db/`: схема (conversations, messages, speakers, settings), миграции, репозитории (CRUD разговоров/сообщений, get/set настроек, session-id на разговор).
- better-sqlite3, файл БД в `app.getPath('userData')`.
- **Тесты:** unit-тесты репозиториев на in-memory SQLite (`:memory:`): создание/чтение/удаление разговора, добавление сообщений, персист настроек, каскадное удаление.
- **Гейт:** тесты зелёные.

## Шаг 3 — IPC-скелет + preload
- `src/preload/index.ts`: типобезопасный `window.api` (contextBridge), только нужные каналы.
- `src/main/ipc/`: хендлеры, дергающие репозитории (шаг 2) и заглушки пайплайна.
- `src/shared/ipc.ts`: контракт каналов и типов запрос/ответ (единый источник для main и preload).
- **Тесты:** unit-тесты хендлеров (вызов репозитория с моками), проверка сериализуемости payload.
- **Гейт:** тесты зелёные; из renderer через devtools `window.api` доступен.

## Шаг 4 — UI-оболочка (пиксель-в-пиксель, статика)
- Перенос дизайн-токенов и вёрстки из `Голосовой чат.dc.html`: сайдбар, шапка+бейдж, лента сообщений (пузыри, chips спикеров), live-блок, голосовая панель (4 состояния), модалка настроек, анимации (волна 28 столбиков, эквалайзер, dots, blink).
- Manrope, все цвета/радиусы/отступы по README «Design Tokens».
- Компоненты рендерятся от props/состояния (пока моковые данные).
- **Тесты:** RTL-тесты рендера ключевых состояний (idle/listening/thinking/speaking), наличие chip-меток, открытие/закрытие модалки, клик по оверлею закрывает.
- **Гейт:** визуальное сравнение с прототипом (скриншот через /run), тесты зелёные.

## Шаг 5 — Связка UI ↔ машина состояний ↔ БД (мок-пайплайн)
- Стор renderer (zustand или useReducer) на базе машины состояний шага 1.
- Реальные разговоры/сообщения/настройки из SQLite через IPC.
- Мок-пайплайн: имитация роста live-транскрипта и ответа (как в прототипе) — чтобы весь UX-цикл работал без ML.
- Поведение: «+ Новый», переключение разговоров, Enter отправляет как «Спикер 1», автоскролл, тумблер диаризации → метки «Вы».
- **Тесты:** интеграционные тесты стора (диспатч событий → корректные состояния и запросы к api-моку); тест автоскролла-триггера.
- **Гейт:** полный UX-цикл кликается на моке; история сохраняется между перезапусками.

## Шаг 6 — Аудиозахват (renderer)
- `getUserMedia` + AudioWorklet: ресемпл в 16 kHz mono PCM (Float32→Int16), чанки по ~ (напр.) 250 мс, передача в main по IPC.
- Выбор микрофона из настроек (enumerateDevices).
- **Тесты:** unit-тесты ресемплера/конвертера PCM (детерминированные буферы), тест чанкинга.
- **Гейт:** при записи в main приходят чанки корректного формата (лог/скрин); тесты зелёные.

## Шаг 7 — Whisper STT (реальный, локальный)
- `src/main/stt/`: обёртка над whisper.cpp (`smart-whisper` или `nodejs-whisper`), стриминговое распознавание по буферу чанков, частичные гипотезы → live-блок.
- Авто-скачивание модели (`large-v3-turbo`, настраиваемо turbo/medium/small) при первом запуске с прогрессом; путь `models/`.
- Язык распознавания: ru по умолчанию.
- Абстракция `SttEngine` (интерфейс), чтобы мокать в тестах.
- **Тесты:** unit-тест форматирования частичных/финальных результатов; тест логики выбора/наличия модели (мок ФС); интеграционный тест распознавания короткого WAV-сэмпла (ru) → ожидаемый текст (толерантно, по ключевым словам).
- **Гейт:** реальная речь с микрофона распознаётся в live-блоке; финальный транскрипт фиксируется; тесты зелёные.

## Шаг 8 — Claude Code CLI (реальный)
- `src/main/claude/claudeCli.ts`: `child_process.spawn('claude', ['-p', text, '--output-format', 'stream-json', '--resume', sessionId])`, парсинг stream-json, стрим токенов в чат.
- session-id на разговор (сохранение/переиспользование через БД), выбор модели из настроек.
- Форматирование промпта из сегментов: `[Спикер N]: …`.
- Graceful-ошибки: CLI не найден / не залогинен → понятная ошибка + подсказка `claude login`.
- Абстракция `LlmClient` для моков.
- **Тесты:** unit-тест парсера stream-json (фикстуры строк), тест сборки промпта из сегментов, тест обработки ошибок (мок spawn: нет бинаря / ненулевой код / not-logged-in). Интеграционный тест против реального `claude -p` (короткий запрос → непустой ответ) — помечен как integration, запускается при доступном CLI.
- **Гейт:** реальный ответ Claude стримится в чат; ошибки отображаются понятно; тесты зелёные.

## Шаг 9 — Настройки, ошибки, первый запуск
- Модалка настроек читает/пишет в БД (модель Claude, модель Whisper, диаризация, голос TTS, микрофон).
- Экран/тосты ошибок: нет CLI, не залогинен, нет модели Whisper (предложить скачать) с прогрессом.
- Удаление разговора.
- **Тесты:** тесты персиста настроек end-to-end (UI → IPC → БД → перезапуск), тест сценариев ошибок.
- **Гейт:** настройки применяются и сохраняются между запусками; сценарии ошибок отрабатывают.

## Шаг 10 — Заглушки диаризации и TTS (чистые интерфейсы)
- `DiarizationEngine` (заглушка: один спикер / «Вы»), `TtsEngine` (заглушка: состояние speaking без реального звука или простой beep).
- Тумблер диаризации управляет метками.
- **Тесты:** контрактные тесты интерфейсов на заглушках.
- **Гейт:** интерфейсы готовы к подключению sherpa-onnx / Piper на следующем этапе.

- **Гейт:** собранное приложение проходит полный сценарий.
